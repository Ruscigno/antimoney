# Infra CLAUDE.md

Terraform configuration deploying Antimoney to GCP. Targets the GCP Always Free tier where possible.

## Architecture

```
Internet
  ├── Cloud Run: antimoney-frontend  (nginx + React SPA)
  └── Cloud Run: antimoney-backend   (Go API)
                     │ VPC Direct Egress (private ranges only)
                     └── Compute Engine VM: antimoney-db
                              │
                         Docker: postgres:16-alpine
                              │ /opt/postgres_data (persistent volume on pd-standard disk)
```

- **Database** runs in a Docker container on an `e2-micro` Compute Engine VM (Always Free in `us-central1`).
- **Backend and frontend** run on Cloud Run (serverless, scales to zero).
- Cloud Run reaches the DB via **VPC Direct Egress** — private IP only, no public exposure of Postgres.
- Artifact Registry stores Docker images with cleanup: untagged images deleted, only last 5 tagged versions kept.

## Free Tier Constraints & Cost-Security Balance

The DB VM must stay on `e2-micro` in `us-central1`, `us-east1`, or `us-west1` and use a `pd-standard` boot disk ≤ 30 GB to remain Always Free eligible. Do not change `machine_type`, `region`, or `disk.type` without understanding the billing impact.

- **Cost vs. Security Priority**: Deploy the project to Google Cloud using the free tier as much as possible without compromising too much security. Maintain a good balance between security and cost, but always try to use free-tier services first.
- **Paid Services Approval**: Paid services are allowed *only* if there are no other viable free solutions. Any introduction of a paid service must be approved beforehand through an exclusive, dedicated git commit.

## Security & Architecture Best Practices

- **Network Isolation**: Ensure Postgres is never exposed to the public internet. Cloud Run reaches the DB via VPC Direct Egress (private IP only).
- **IAM & Service Accounts**: Cloud Run services and DB VMs should run as dedicated Service Accounts following the principle of least privilege, rather than default compute service accounts.
- **WAF & DDoS Protection**: Consider placing Cloud Run behind a Global Load Balancer with Cloud Armor (WAF) to defend against DDoS attacks, rate-limit abuse, and common web exploits.
- **Secret Management**: Currently, passwords live in Terraform state. For improved security, consider moving to Google Secret Manager for injecting production secrets (e.g., `JWT_SECRET`, `DATABASE_URL`) directly into Cloud Run at runtime.

## Key Design Decisions

- **Terraform manages infrastructure; `deploy.sh` manages image updates.** Cloud Run service image is set to a placeholder on initial `terraform apply`. The `lifecycle.ignore_changes` block on `image` prevents Terraform from reverting deployments done outside of TF.
- **Passwords are generated by Terraform** (`random_password`). They live only in `terraform.tfstate` — never hardcode them. The DB password is injected into Cloud Run via `DATABASE_URL` env var at resource creation time.
- **No Cloud SQL.** Using a self-managed Postgres on a VM avoids Cloud SQL costs. Tradeoff: no automated backups, no HA.
- **Build staging bucket** auto-deletes objects after 7 days and has soft-delete disabled to avoid storage costs.

## State Files

`terraform.tfstate` and `terraform.tfstate.backup` are **local state** — they contain secrets (generated passwords). Do not commit them to git. If you need to share state across machines, migrate to a GCS backend.

## Variables

| Variable | Default | Notes |
|---|---|---|
| `project_id` | — | Required; no default |
| `region` | `us-central1` | Must be a free-tier eligible region for the VM |
| `zone` | `us-central1-a` | Must match region |
| `db_user` | `antimoney` | Postgres username |
| `db_name` | `antimoney` | Postgres database name |

Set `project_id` via `terraform.tfvars` (not committed) or `-var project_id=...`.

## Outputs

After `terraform apply`:

```bash
terraform output frontend_url        # The live app URL
terraform output backend_url         # Backend Cloud Run URL
terraform output artifact_registry   # Docker registry path for image pushes
terraform output db_internal_ip      # Internal IP of the DB VM (for deploy.sh)
terraform output build_staging_bucket
```

## Common Operations

```bash
# Initial setup
terraform init
terraform plan -var project_id=YOUR_PROJECT
terraform apply -var project_id=YOUR_PROJECT

# After infrastructure exists, deploy a new image via deploy.sh (not terraform)
# Terraform will not touch the Cloud Run image after initial deploy (ignore_changes)

# Destroy everything (WARNING: destroys DB VM and all data)
terraform destroy -var project_id=YOUR_PROJECT
```

## Gotchas

- **Database has no automated backups.** The data lives on the VM's persistent disk at `/opt/postgres_data`. If the VM is destroyed, data is lost. `terraform destroy` will destroy it.
- **Ephemeral external IP on the DB VM** is required only during startup to pull Docker/packages. After initial boot it's not strictly needed, but the current config keeps it. This does not expose Postgres to the internet (firewall only allows `10.0.0.0/8`).
- **Cloud Run timeout is 30 seconds** (set in Go middleware). Long imports/exports must complete within this window.
- **`disable_on_destroy = false`** on enabled APIs means `terraform destroy` will not disable GCP APIs — intentional, to avoid accidentally disabling shared project APIs.
- **`force_destroy = true`** on the build staging bucket means `terraform destroy` will delete it even if it contains objects.
